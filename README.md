# hb_compile

**English** | [Português (Brasil)](README.pt-BR.md)

Build orchestration layer for compiling Harbour from a local checkout. If the
source tree does not exist, the project clones `https://github.com/harbour/core.git`
into `scratch\harbour-core` by default.

The repository keeps source checkouts, downloaded tools, logs, and installation
outputs separate from the Harbour source tree.

## Layout

- `config\profiles.json`: build profiles and the default Harbour source.
- `config\dependencies.json`: optional dependency catalog used by `-Full`.
- `config\docker\linux\`: reproducible Linux Docker images.
- `scripts\Invoke-HarbourBuild.ps1`: main build runner.
- `scripts\Bootstrap-Tools.ps1`: local Zig and MinGW-w64 provisioning.
- `scripts\Resolve-HarbourDeps.ps1`: optional dependency resolver.
- `build-sandbox.ps1`: isolated portable-toolchain builds in Windows Sandbox.
- `out\`: installations generated by each profile.
- `logs\`: complete build logs.
- `tools\`: downloaded local toolchains.
- `scratch\`: source checkouts and temporary data.

See [HOWTO.md](HOWTO.md) for the complete workflow,
[DEPENDENCIES.md](DEPENDENCIES.md) for optional libraries, and
[OPENADS.md](OPENADS.md) for the OpenADS fallback. Windows Sandbox behavior and
limitations are documented in [BUILD-SANDBOX.md](BUILD-SANDBOX.md).

## Quick start

```powershell
.\scripts\Show-Status.ps1
.\scripts\Invoke-HarbourBuild.ps1 -ListProfiles
.\build-zig.ps1 -DryRun
.\build-zig.ps1 -Clean
```

Run an isolated Windows Sandbox build:

```powershell
.\build-sandbox.ps1 -BuildProfile zig -Clean -SandboxTimeoutMinutes 60
```

The Sandbox launcher is intended for clean-environment and reproducibility
validation, not as the primary development build path. Local wrappers are much
faster for routine and incremental builds. See [BUILD-SANDBOX.md](BUILD-SANDBOX.md).
Use `-LocalWorkspace` to keep build I/O on the guest disk when the repository is
stored on unreliable external or network storage.

Validate an installed build with `samples\hello.prg`:

```powershell
.\scripts\Test-HarbourInstall.ps1 -Profile zig
```

## Harbour source

The defaults are stored in `config\profiles.json`:

```json
{
  "harbourRepository": "https://github.com/harbour/core.git",
  "harbourRoot": "scratch\\harbour-core"
}
```

Override the repository, checkout directory, or initial ref:

```powershell
.\build-zig.ps1 -HarbourRepository https://github.com/user/core.git -HarbourRoot scratch\harbour-fork
.\build-zig.ps1 -HarbourRef my-branch
```

When `HarbourRoot` already exists, the runner does not change its checked-out
ref. Use `git -C <HarbourRoot> checkout <ref>` explicitly.

## Build profiles

### Standard, MSVC, and MinGW-w64

`standard` lets Harbour's `win-make.exe` detect the compiler from the current
environment. `msvc64` loads Visual Studio Build Tools through `vswhere` unless
`-NoVsDevShell` is used. `mingw64` accepts `-MinGwPath`, reuses
`tools\mingw64`, checks `PATH`, and can bootstrap a portable toolchain.

```powershell
.\build-standard.ps1 -Clean
.\build-msvc64.ps1 -Clean
.\build-mingw64.ps1 -MinGwPath C:\Tools\mingw64\bin -Clean
```

### Cygwin and MSYS2 MSYS

These profiles run `make` inside their Bash environments and use system
packages instead of Windows vcpkg dependencies.

```powershell
.\build-cygwin.ps1 -CygwinBash C:\cygwin64\bin\bash.exe -Clean
.\build-msys.ps1 -MsysBash C:\msys64\usr\bin\bash.exe -Clean
```

### WSL and Docker

```powershell
.\build-linux-wsl.ps1 -WslDistro Ubuntu-24.04 -Clean
.\build-linux-docker.ps1 -Clean
```

WSL requires build packages in the selected distribution. Docker builds its
image from `config\docker\linux\Dockerfile`; use `-SkipDockerBuild` to reuse an
existing image.

### Zig

Zig profiles support native Windows builds and Windows/Linux cross-builds:

```text
zig                  native Windows
zig-win64-gnu        Windows x64 GNU target
zig-win64-msvc       Windows x64 MSVC target
zig-win32-gnu        Windows x86 cross-build
zig-win-arm64        Windows ARM64 cross-build
zig-linux-x64        Linux x64 cross-build
zig-linux-arm64      Linux ARM64 cross-build
```

Cross-build profiles use a native build as `HB_HOST_BIN` when host tools are
required. Tool lookup order is: explicit path, `tools\<compiler>`, valid
`PATH`, then automatic bootstrap. Use `-SkipToolBootstrap` to disable downloads.

## Full builds

`-Full` resolves optional contrib dependencies. Windows profiles use vcpkg;
Cygwin, MSYS, WSL, and Docker use packages installed in their environment.

```powershell
.\build-full-zig.ps1 -IgnoreDependency qt -Clean
.\build-full-msvc64.ps1 -DependencySet database -Clean
.\build-full-linux-docker.ps1 -IgnoreDependency qt -Clean
```

Useful controls include `-DependencySet`, `-Dependency`, `-IgnoreDependency`,
`-SkipDependencyInstall`, and `-StrictDependencies`. See
[DEPENDENCIES.md](DEPENDENCIES.md).

## Outputs and diagnostics

Each run prints the source root, profile, runner, install directory, log path,
command, and Harbour environment variables. Installations go to `out\<profile>`
and timestamped logs go to `logs\`.

`-DryRun` validates and prints the intended command but does not compile or
replace artifacts. An existing installation can therefore still be found after
a successful dry run; check log and artifact timestamps before treating it as a
new build.
